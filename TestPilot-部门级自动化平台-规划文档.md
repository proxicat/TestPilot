## 0. 立场与判断(不粉饰)

当前实现已经把**最难、最新的内核**跑通并在真实站点验证:AI 视觉规划 → 生成 → 执行(注入钱包/fork/真实上链)→ 缓存(~3×)→ 持久化 → 多项目 → 端到端 UI 串联 → 可导出 Playwright+Midscene 工程。

但"接管一个部门"靠的不是内核,而是内核外面那圈 **QA 工程学**:**oracle、基线、测试数据、报告深度、追溯、规模化、flake 治理**。这圈目前基本是空的。

**成熟度现状(接管部门 = 参照满分):**

| 维度 | 现状 | 评级 |
|---|---|---|
| AI 执行内核(视觉/钱包/fork/缓存) | 跑通并验证 | 🟢 强 |
| 用例**发现** | AI 规划 + 深度爬取 | 🟢 良 |
| 用例**设计质量 / oracle** | 弱断言,无方法论 | 🔴 差 |
| **基线**(视觉/性能/功能) | 无 | 🔴 缺 |
| **报告深度**(Midscene 全量) | 只用一小片 | 🔴 差 |
| **测试数据 / 环境 / 密管** | 基本无,凭据硬编码 | 🔴 差 |
| 追溯 / 覆盖 / 缺陷联动 | 无 | 🔴 缺 |
| 规模化(并发/CI/调度/flake/自愈) | 无 | 🔴 缺 |
| 持久化 / 多项目 / UI 串联 | 已具备 | 🟢 良 |

**结论**:当前 ≈ **可演示 POC / 早期 alpha**。本规划的任务:把 🔴 逐个变 🟢。

## 1. 目标定位与价值主张

**接管一个部门的自动化测试业务** = 平台需要覆盖 QA 团队的完整价值链:

```
需求/PRD → 测试设计 → 用例规格 → 测试数据/环境 → 执行 → 断言/基线比对
        → 缺陷定位 → 报告/趋势 → 回归维护 → 覆盖度量 → 决策
```

我们的差异化(相对 testRigor/mabl/Octomind 等):**自建视觉模型(数据不出内网)+ AI 规划 + 视觉驱动无 selector 维护 + dapp/Web3 能力**。但要"接管",必须先补齐上面这条链里 🔴 的环节,否则只是"能生成用例的玩具"。

**北极星指标(判断是否真能接管):**
- **回归缺陷捕获率** ≥ 人工回归水平(基线是核心)
- **用例维护成本** 相比脚本化下降 ≥ 60%(靠视觉 + 自愈)
- **Flake 率** < 2%
- **失败定位时间(MTTR)** < 5 分钟(靠 Midscene 全量报告)
- **回归自动化覆盖** ≥ 部门手工回归的 80%

## 3. 直接回应六个专业问题(现状 → 目标)

### 3.1 提示词与规划方法论
- **现状**:单轮、无 schema、无方法论的 `EXPLORE_PROMPT` / `generateCode`。产出"合理冒烟清单"。
- **目标**:提示词工程化 —— 注入 **PRD/需求文档 + 页面 DOM/截图 + 领域词表**;强制模型走**测试设计方法论**(等价类、边界值、判定表、状态迁移、负面/异常路径);用 **JSON Schema / 强类型输出**约束;多轮(发现→分类→细化→评审)。

### 3.2 用例金标准
- **现状**:`title/priority/reason/steps`,oracle 泛化,无数据/追溯/后置。
- **目标**:用例规格补齐 **ID · 需求追溯 · 前置条件 · 测试数据(引用而非内联)· 步骤 · 每步预期 · 后置/清理 · 类型(功能/负面/边界)· 独立可重复**。

### 3.3 代码生成与框架
- **现状**:导出已是 Playwright + `PlaywrightAiFixture`(框架化 ✅),但浅(无 POM/fixtures/参数化/tag/重试)。
- **目标**:导出工程升级 —— Page Object 可选、fixtures/setup-teardown、**数据驱动参数化**、tag/分组、重试/超时策略、CI 配置、断言库。

### 3.4 数据管理
- **现状**:**凭据从页面就地嵌进步骤**(缓存里可见 `standard_user/secret_sauce`),无密管/环境/数据工厂。
- **目标**:**Secrets 保险库 + 账号池(多角色)+ 环境配置(dev/staging/prod)+ 测试数据工厂/fixtures + 数据 setup/teardown + 参数化**。步骤里用 `${env.user}`/`${secret.pwd}` 占位,运行时注入。

### 3.5 报告完整性(Midscene 全量)
- **现状**:只存 `status/duration/自拼文本/自截图`;**Midscene 的 3–4MB 交互式 HTML 报告 + AI 推理 + `ai-profile-stats`(token/成本/耗时)+ 带元素标注的截图 + dump 全丢了**。
- **目标**:每次运行**落盘并索引** Midscene 的 `report/*.html`、`log/*`、`dump/*`;Runs 页**内嵌富报告**、展示 AI 推理链、token/成本、标注截图、时间线。这是低成本、高说服力的一步。

### 3.6 基线(视觉/性能/功能)
- **现状**:**完全没有** → 没有回归检测能力。
- **目标**:
  - **视觉基线**:每步/关键页截图入基线库,像素+感知 diff,差异审批(approve/reject)工作流,忽略区域。
  - **性能基线**:采集 Playwright trace / Web Vitals(LCP/CLS/TBT)/ 关键时序,设**预算阈值**,超标即失败。
  - **功能基线**:黄金输出/契约(如 `aiQuery` 提取的结构化结果比对),把 `aiAssert` 从"流程完成"升级为明确断言。

## 4. 分阶段路线图

### Phase 0 — AI 执行内核(✅ 已完成)
explore→cases→run→export · 持久化 · 多项目 · 钱包/dapp/fork · 缓存 · UI 串联。

### Phase 1 — 可信回归(最高优先,平台立身之本)
> 目标:让平台**具备回归检测能力**并**让失败可定位**。

- **1a 接管 Midscene 全量报告**:运行产物落盘(report/log/dump),DB `runs` 增 `reportPath/dumpPath/tokens/costUSD`;Runs 页内嵌 HTML 报告 + AI 推理 + 成本 + 标注截图。
- **1b 视觉基线**:基线库(首跑建基线,后续 diff);差异审批 UI;`baselines` 表 + 图存储。
- **1c 性能基线**:采集 trace/Web Vitals + 预算;超标标红。
- **1d 强 oracle**:用例增"预期结果"结构;生成的 `aiAssert` 用明确断言;支持 `aiQuery` 结构化比对。
- **交付验收**:同一用例改动前后能**自动报出视觉/性能/功能回归**;失败一键看 Midscene 报告定位。

### Phase 2 — 测试资产工程化
> 目标:用例/数据达到可交付、可复用、可安全运行。

- **2a 用例金标准 schema**(追溯/前置/数据/每步预期/后置/类型)。
- **2b 测试数据 & 环境 & 密管**(保险库/账号池/env 配置/数据工厂/参数化)。
- **2c 规划方法论**(PRD 输入 + 等价类/边界/负面 + JSON Schema 输出 + 多轮评审)。
- **2d 导出工程升级**(POM/fixtures/参数化/tag/重试/CI 配置)。

### Phase 3 — 规模化与可信度
> 目标:能承载部门级并发回归,结果可信。

- **3a 并发执行队列**(worker 池 + 模型排队,`Run all P0` 走队列 + 汇总)。
- **3b CI/CD 集成 + 定时调度**(GitHub Actions/Jenkins webhook;cron)。
- **3c Flake 治理**(自动重试 + flake 检测 + 隔离/quarantine + 稳定度评分)。
- **3d 自愈**(selector/视觉漂移 → Midscene 缓存失效 → 自动重规划;失败诊断分类)。
- **3e 趋势看板**(通过率/flake/MTTR/覆盖随时间)。

### Phase 4 — 运营与集成(接管落地)
- 缺陷系统联动(Jira 自动建单,附报告/截图)· Slack 通知 · RBAC/审计/多租户 · 需求覆盖度量 · 跨浏览器/设备 · 可访问性 · API/契约测试 · 成本治理(token 预算/模型路由)。

## 5. 数据模型演进(支撑上述)

新增/扩展(在现有 `projects/test_cases/runs` 基础上):

```
environments   (id, projectId, name, baseUrl, varsJson)          # dev/staging/prod
secrets_ref    (id, projectId, key, envId)                       # 只存引用,值在保险库
test_data      (id, projectId, name, kind, dataJson)             # fixtures/账号池/数据集
test_cases  += (requirementId, preconditions, expected, postSteps, type, dataRef, envRef)
baselines      (id, caseId, kind[visual|perf|functional], key, ref, status, approvedBy, createdAt)
run_artifacts  (id, runId, kind[report|dump|trace|screenshot], path, meta)
runs        += (reportPath, tokens, costUSD, env, commitSha, triggeredBy, flaky)
run_steps      (id, runId, idx, action, expected, status, screenshotRef, aiReasoning, ms)
requirements   (id, projectId, extId, title)                     # 追溯 + 覆盖
flakiness      (caseId, window, failRate, verdict)
```

> 生产建议:SQLite → **PostgreSQL**;制品(报告/截图/trace)→ **对象存储**;secrets → **Vault/KMS**。

## 7. 风险与依赖

| 风险 | 影响 | 应对 |
|---|---|---|
| 自建 VL 模型慢(35B,~15s/步) | 批量回归耗时 | 缓存(已有)+ 并发队列 + 更小 VL 模型 + 结果复用 |
| 视觉基线易受动态内容干扰 | 误报 | 忽略区域 + 感知 diff 阈值 + AI 语义比对兜底 |
| AI 断言不稳定/幻觉 | 假绿/假红 | 明确 oracle + 多次投票 + 关键步用确定性断言 |
| 反爬墙(Cloudflare) | 探索/执行受阻 | 会话复用 + 受控环境 + 与被测方约定测试通道 |
| 凭据/密管合规 | 安全事故 | Vault/KMS + 最小权限 + 审计(P2 必做) |

## 9. 起步建议(我的专业推荐)

**从 M1 开始**——即 **P1a 接管 Midscene 全量报告** + **P1b 视觉基线**。理由:
1. **回归检测**是"自动化测试"的立身之本,基线是它的前提;当前恰恰缺。
2. **Midscene 报告接管**成本最低、说服力最高(数据已经产出,只是没接),能立刻让平台"专业感"到位、失败可定位。
3. 两者叠加,能在真实站点上演示"改动 → 自动报出视觉/功能回归 → 一键定位",这是打动一个 QA 部门的关键一击。
